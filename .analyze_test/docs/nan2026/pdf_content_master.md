# 문서 1. Bee House 게임 소개 및 설명

## 2. 한 줄 소개

색의 흐름을 읽고 육각 보드에서 가장 좋은 경로를 찾는 60초 드래그 퍼즐.

### 한 줄 소개 대안

상단의 색 순서를 육각 보드 위 경로로 재현하며 기록을 경쟁하는 타임어택 퍼즐.

### 짧은 홍보 문구 대안

색을 보지 말고, 색의 흐름을 읽어라.

## 3. 게임 개요

Bee House는 화면 상단에 제시되는 컬러 바의 색 순서를 읽고, 같은 순서의 육각 타일을
보드 위에서 찾아 드래그로 연결하는 타임어택 퍼즐 게임이다.

일반적인 매치 퍼즐은 같은 색을 모으거나 정해진 개수의 블록을 제거하는 방식이 많다.
Bee House는 단순히 같은 색을 찾는 데서 끝나지 않는다. 플레이어는 컬러 바 안에서
사용 가능한 연속 구간을 먼저 발견하고, 그 순서를 유지하면서 보드 위 인접 타일을
경로로 만들어야 한다. 따라서 색을 인식하는 능력, 순서를 기억하는 능력, 공간에서
경로를 탐색하는 능력, 제한 시간 안에 판단하는 능력이 동시에 요구된다.

게임은 한 손가락 드래그만으로 조작할 수 있도록 설계했다. 규칙은 짧은 시간 안에
이해할 수 있지만, 어떤 구간을 선택하고 어떤 경로를 연결할지에 따라 점수와 남은
시간이 크게 달라진다. 배우기는 쉽고 기록을 높이기 위해서는 반복적인 판단과 숙련이
필요한 구조가 Bee House의 핵심이다.

## 4. 기획 의도

Bee House의 시작점은 “색 맞추기 퍼즐에서 같은 색이 아니라 색의 순서 자체를
플레이하게 만들 수 있을까?”라는 질문이었다.

육각 보드는 사각형 보드보다 한 타일에서 이동할 수 있는 방향이 많다. 이 특성을
활용하면 짧은 규칙 안에서도 다양한 경로가 만들어진다. 플레이어는 당장 성공할 수
있는 짧은 경로를 선택할지, 더 큰 보너스를 얻을 수 있는 긴 경로를 찾을지 결정해야
한다. 여기에 시간 보너스와 콤보를 결합해 한 번의 정답을 찾는 퍼즐이 아니라, 빠른
판단을 계속 이어가는 흐름 중심의 게임을 만들고자 했다.

또한 한 판의 시간이 짧아 언제든 부담 없이 시작할 수 있으면서도, 오늘의 퍼즐과
랭킹을 통해 매일 다시 플레이할 이유가 생기도록 구성했다.

## 5. 핵심 차별점

### 5.1 같은 색이 아니라 색의 순서를 연결한다

플레이어는 상단 컬러 바에서 길이 3 이상의 연속 구간을 선택한다. 이후 보드에서 해당
색 순서를 그대로 재현하는 인접 경로를 만들어야 한다. 같은 색 타일을 많이 모으는
기존 방식과 달리, 순서가 달라지면 같은 색 조합이라도 성공하지 않는다.

### 5.2 육각 보드가 만드는 경로 탐색

보드는 7행 6열의 육각 타일로 구성된다. 육각 구조에서는 여러 방향으로 경로를
확장할 수 있기 때문에 같은 컬러 바를 보고도 플레이어마다 서로 다른 경로를 선택할
수 있다. 짧고 안전한 경로와 길고 높은 점수를 주는 경로 사이에서 즉각적인 판단이
필요하다.

### 5.3 성공이 다음 플레이 시간을 만든다

기본 제한 시간은 60초다. 매칭에 성공하면 경로 길이에 따라 시간이 추가된다. 좋은
경로를 빠르게 찾는 플레이어는 제한 시간을 연장하며 더 오래 플레이할 수 있다.
시간이 단순히 줄어드는 자원이 아니라 실력으로 다시 획득할 수 있는 보상이 된다.

### 5.4 짧은 규칙과 장기적인 기록 경쟁

게임 조작은 드래그 하나로 끝나지만, 점수 보너스, 콤보, 오늘의 퍼즐, 주간 랭킹,
전체 랭킹, 미션, 시즌 티어가 장기적인 목표를 제공한다. 한 판은 짧지만 기록을
갱신하기 위해 반복해서 플레이할 수 있다.

## 6. 플레이 목표

제한 시간 안에 컬러 바와 일치하는 경로를 최대한 많이 연결해 높은 점수를 얻는 것이
기본 목표다.

플레이어는 다음 요소를 함께 고려해야 한다.

- 현재 컬러 바에서 사용할 수 있는 연속 색 구간
- 보드에서 실제로 만들 수 있는 인접 경로
- 짧은 경로를 빠르게 연결할지, 긴 경로를 탐색할지
- 이전 성공 후 4초 이내에 다음 성공을 이어 콤보를 유지할 수 있는지
- 남은 시간과 현재 점수

## 7. 조작 방법

1. 화면 상단 컬러 바를 확인한다.
2. 컬러 바 안에서 길이 3 이상의 연속된 색 구간을 찾는다.
3. 보드에서 첫 번째 색과 같은 육각 타일을 누른다.
4. 손가락을 떼지 않고 인접한 타일을 색 순서대로 드래그한다.
5. 같은 드래그 안에서 이미 지나간 타일은 다시 사용할 수 없다.
6. 컬러 바의 연속 구간과 정확히 일치하는 경로를 완성하면 매칭에 성공한다.
7. 성공한 타일은 사라지고 새로운 타일이 보드에 채워진다.
8. 컬러 바에서도 사용한 구간이 제거되고 새로운 색이 뒤에 추가된다.

잘못된 순서로 연결하거나 유효한 컬러 바 구간과 일치하지 않는 경로를 만들면
매칭으로 처리되지 않는다. 게임은 드래그 중 현재 경로의 상태를 시각적으로 표시해
플레이어가 성공 가능 여부를 빠르게 이해하도록 돕는다.

## 8. 점수 시스템

### 8.1 기본 점수

길이 3 이상의 유효한 경로를 완성하면 기본 100점을 얻는다.

### 8.2 길이 보너스

3개를 초과해 연결한 타일 수가 많을수록 보너스가 빠르게 증가한다.

- 추가 타일 수: `n = 경로 길이 - 3`
- 길이 보너스: `(n × 150) + (n² × 50)`

따라서 단순히 성공 횟수를 늘리는 것뿐 아니라, 가능한 상황에서 더 긴 경로를 찾는
것이 고득점에 중요하다.

### 8.3 콤보 보너스

이전 매칭 성공 후 4초 이내에 다음 매칭에 성공하면 콤보가 이어진다.

- 콤보 보너스: `(현재 콤보 횟수 - 1) × 50`

콤보는 플레이어가 보드를 천천히 한 번씩 해결하는 대신, 다음 경로까지 미리 읽으며
빠르게 플레이하도록 유도한다.

## 9. 시간 시스템

- 기본 제한 시간: 60초
- 매칭 성공 시 경로 길이에 따라 시간 추가
- 시간 보너스: `1.4 + (경로 길이 - 2) × 0.65초`
- 추가 시간은 정해진 상한 안에서 누적

길이가 긴 경로는 점수뿐 아니라 생존 시간도 늘려준다. 플레이어는 긴 경로를 찾는 데
시간을 더 쓸지, 짧은 경로를 빠르게 연결해 안정적으로 시간을 확보할지 판단해야 한다.

## 10. 게임 종료 조건

다음 중 하나가 발생하면 게임이 종료된다.

1. 남은 시간이 0초가 된 경우
2. 현재 보드에서 컬러 바와 일치하는 유효한 경로를 더 이상 만들 수 없는 경우

게임 종료 후에는 최종 점수, 최대 콤보, 가장 긴 경로, 매칭 횟수와 같은 결과를
확인할 수 있다. 플레이 기록에 따라 다시 플레이하거나 리플레이를 확인하고 결과를
공유할 수 있다.

## 11. 주요 게임 모드와 기능

### 11.1 일반 플레이

매번 새로운 시드로 보드가 생성되는 기본 모드다. 제한 시간 동안 최대한 높은 점수를
얻어 개인 최고 기록과 전체 랭킹에 도전한다.

### 11.2 오늘의 퍼즐

날짜를 기준으로 정해진 시드를 사용해 모든 플레이어에게 같은 초기 조건을 제공한다.
무작위성이 있는 게임에서도 동일한 시작점을 제공하기 때문에 결과를 비교하기 쉽고,
매일 새로운 경쟁 목표가 생긴다.

오늘의 퍼즐은 공식 점수 제출과 연습 흐름을 구분한다. 네트워크나 제출 과정에서
문제가 발생하더라도 완료되지 않은 시도를 복구할 수 있도록 상태를 관리한다.

### 11.3 랭킹

- 전체 최고 기록 랭킹
- 주간 기록 랭킹
- 오늘의 퍼즐 랭킹

로그인한 사용자는 서버에 점수를 동기화하고 다른 플레이어와 기록을 비교할 수 있다.

### 11.4 시즌 티어

점수와 주간 기록을 바탕으로 플레이어의 진행 상태를 시각적으로 보여준다. 상위
구간에는 Master와 Challenger, 500,000점 이상의 Jesus 티어가 포함된다. 단순한 순위 숫자 외에도 장기적으로
도전할 수 있는 목표를 제공한다.

### 11.5 리플레이

게임 중 성공한 이동 경로와 콤보 정보를 기록한다. 리플레이를 실행하면 같은 초기
시드에서 각 이동 경로를 순서대로 재현한다. 결과 화면의 숫자만 보여주는 것이 아니라,
어떤 판단과 경로가 점수로 이어졌는지 다시 볼 수 있다.

### 11.6 결과 공유

플레이 결과를 이미지 형태로 공유할 수 있으며, 리플레이 코드를 받은 사용자는 설정에서
코드를 불러와 같은 플레이를 재생할 수 있다.

### 11.7 다국어 지원

한국어와 영어를 지원한다. 설정 화면에서 언어를 변경할 수 있으며, 게임의 핵심
조작이 텍스트보다 색과 드래그 중심이기 때문에 언어에 관계없이 규칙을 이해하기 쉽다.

## 12. 사용자 경험 설계

Bee House는 네오 브루탈리즘을 기반으로 굵은 테두리, 선명한 색, 평면 하드 섀도우를
사용한다. 장식적인 효과보다 현재 점수, 남은 시간, 컬러 바, 보드와 주요 행동 버튼이
분명하게 보이는 것을 우선했다.

드래그, 성공, 실패, 시간 증가와 게임 종료에는 시각 효과, 효과음과 진동 피드백을
적용한다. 한 손가락으로 조작하는 짧은 게임에서 각 입력이 정확히 전달됐다는 감각을
주는 것이 중요하기 때문이다.

휴대폰과 iPad 화면 크기에 따라 레이아웃과 여백을 조정한다. 핵심 정보와 플레이
영역이 화면 밖으로 밀리지 않도록 반응형 UI를 구성했다.

## 13. 기술 구성

### 클라이언트

- Flutter / Dart 단일 코드베이스
- GetX 기반 상태 관리와 의존성 관리
- shared_preferences 기반 로컬 설정과 진행 정보
- audioplayers 기반 BGM 및 효과음
- 햅틱, 공유 시트와 플랫폼 로그인 연동

### 서버

- Supabase Auth 기반 사용자 인증
- PostgreSQL 기반 프로필, 점수와 랭킹 데이터
- RPC 기반 점수 제출과 랭킹 처리
- Row Level Security를 이용한 데이터 접근 제어
- 날짜가 포함된 SQL migration으로 데이터베이스 변경 이력 관리

### 게임 재현성

- 시드 기반 보드 생성
- 오늘의 퍼즐 날짜별 고정 시드
- 플레이 이동 경로 기록
- 동일 시드와 입력 기록을 이용한 리플레이

# 문서 2. Bee House AI 활용 기술 문서

## 1. AI 활용 요약

Bee House는 생성형 AI가 플레이 도중 정답이나 콘텐츠를 실시간으로 만들어주는 게임이
아니다. 대신 사람이 게임의 방향과 품질 기준을 정하고, AI를 코드 분석, 구현,
리팩터링, 장애 시나리오 점검, 테스트와 문서화에 활용한 AI-directed production
프로젝트다.

이 프로젝트에서 AI의 가치는 기능 하나를 추가한 데 있지 않다. 개발자가 해야 할
판단을 대신하는 것이 아니라, 개발자가 더 넓은 범위를 빠르게 확인하고 반복 작업을
검증 가능한 형태로 수행할 수 있도록 제작 과정 전체를 보조했다.

런타임 게임 규칙은 결정론적으로 유지했다. 오늘의 퍼즐은 같은 날짜에 같은 초기
시드를 제공해야 하고, 리플레이는 실제 입력을 같은 순서로 재현해야 한다. 외부 AI
API의 응답에 따라 게임 결과가 달라지지 않기 때문에 랭킹의 공정성과 플레이의
재현성을 유지할 수 있다.

## 2. 사용한 AI 도구

- 도구명: OpenAI Codex
- 활용 범위: 저장소 분석, 코드 수정안 작성, 리팩터링, 버그 원인 분석, 테스트 실행,
  공식 공모전 요건 조사, 제출 체크리스트 작성, 기술 문서 초안 작성
- 사용 방식: 개발자가 목표와 변경 금지 범위를 지시하고, Codex가 관련 파일과 실행
  결과를 확인한 뒤 수정안을 제시하거나 적용
- 최종 결정: 모든 변경의 채택 여부와 게임 정책은 개발자가 결정

`[과거 개발 과정에서 Codex 외 ChatGPT, Claude, Gemini 등 다른 AI를 사용했다면
도구명, 사용 시기와 역할을 사실대로 추가]`

## 3. AI 활용 원칙

### 3.1 사람은 방향을 결정한다

다음 항목은 사람이 직접 정의하고 최종 결정했다.

- 게임의 핵심 아이디어
- 컬러 바와 육각 경로를 결합한 규칙
- 최소 매칭 길이
- 점수와 시간 보너스 정책
- 콤보 제한 시간
- 게임 종료 조건
- 비주얼 방향과 사용자 경험 우선순위
- 출시 여부와 기능 채택 여부

### 3.2 AI는 분석과 반복을 가속한다

AI는 다음 작업을 지원했다.

- 저장소 전체 구조 파악
- 변경과 관련된 파일 탐색
- 큰 문제를 구현 가능한 작은 작업으로 분해
- 코드 수정안과 데이터베이스 migration 작성
- 예외 상황과 실패 시나리오 탐색
- 테스트 케이스 제안 및 실행
- 중복 코드와 큰 파일의 리팩터링
- 모바일 화면 크기별 UI 점검
- 문서와 실제 코드의 불일치 확인
- 공모전 제출 요구사항과 현재 결과물 비교

### 3.3 실행 결과로 검증한다

AI가 만든 결과는 답변 내용만 보고 채택하지 않는다. 실제 diff, 정적 분석, 자동
테스트와 빌드 결과를 확인한다. 서버 변경은 SQL migration으로 기록하고, 게임 규칙은
문서와 코드가 일치하는지 비교한다.

## 4. AI와 협업한 일반적인 작업 흐름

1. 개발자가 해결하려는 문제와 유지해야 할 규칙을 설명한다.
2. AI가 저장소에서 관련 코드, 문서와 테스트를 찾는다.
3. AI가 현재 동작과 문제 발생 조건을 요약한다.
4. 개발자가 변경 방향이나 금지 범위를 확인한다.
5. AI가 코드 또는 문서 수정안을 작성한다.
6. 변경된 파일의 diff를 확인한다.
7. 정적 분석과 자동 테스트를 실행한다.
8. 필요한 경우 실제 기기에서 사용자가 플레이 감각과 UI를 확인한다.
9. 문제가 있으면 실행 결과를 다시 AI에 제공해 반복 수정한다.
10. 검증된 변경만 커밋 기록으로 남긴다.

## 5. 구체적인 AI 활용 사례

### 5.1 일일 도전 제출 실패 복구

#### 문제

오늘의 퍼즐은 하루에 한 번 공식 기록을 제출하는 흐름을 갖는다. 사용자가 공식 도전을
시작한 뒤 앱 종료, 네트워크 오류 또는 서버 제출 실패가 발생하면, 시도 상태와 점수
제출 상태가 서로 어긋날 수 있다. 단순히 “이미 시도함”으로 잠그면 실제 점수를 제출하지
못한 사용자가 다시 진행할 수 없고, 무조건 재시도를 허용하면 하루 한 번이라는 규칙이
깨질 수 있다.

#### AI 활용

AI와 함께 일일 도전의 상태를 다음 관점에서 나눠 분석했다.

- 공식 도전을 시작했는가
- 게임을 정상적으로 끝냈는가
- 점수를 서버에 제출했는가
- 로컬에 제출 대기 데이터가 남아 있는가
- 앱을 다시 실행했을 때 어떤 상태를 복구해야 하는가

AI는 클라이언트 진입 조건, 로컬 제출 대기 서비스, 서버 RPC와 RLS 정책, 사용자 안내
UI를 함께 검토하고, 완료되지 않은 시도는 복구하되 이미 점수가 확정된 시도는 다시
제출하지 못하도록 수정안을 만들었다.

#### 결과

- 완료되지 않은 공식 도전의 재진입 처리
- 로컬 제출 대기 정보 관리
- 점수 제출 복구 서비스 추가
- 서버 migration을 통한 재시도 조건 정리
- 이미 점수가 있는 경우와 제출이 끝나지 않은 경우를 분리한 자동 테스트 추가

#### 확인 가능한 근거

- 주요 커밋: `b7c4801`, `9e2c5b7`

- 테스트: `test/daily_challenge_entry_test.dart`
- DB 변경: `supabase/migrations/202604140002_allow_retry_for_incomplete_daily_attempts.sql`

### 5.2 대규모 파일의 책임 단위 리팩터링

#### 문제

개발 초기에는 홈, 랭킹, 설정, 보드와 게임 컨트롤러 코드가 큰 파일에 모여 있었다.
기능이 늘어나면서 한 화면을 수정할 때 관련 없는 코드까지 함께 확인해야 했고, AI에게
작업을 지시할 때도 변경 범위가 지나치게 커질 위험이 있었다.

#### AI 활용

핵심 동작을 바꾸지 않는다는 조건으로 파일을 책임 단위로 나누도록 지시했다. AI는
기존 참조 관계와 상태 흐름을 확인하고 다음 영역을 분리했다.

- 육각 보드 레이아웃, 그리기, 입력 처리
- 게임 컨트롤러의 생성, 매칭, 드래그, 라운드와 튜토리얼
- 홈 화면의 브랜딩, 버튼, 점수와 랭킹 미리보기
- 랭킹 화면의 상태, 목록 아이템과 내 순위 카드
- 설정 화면의 헤더, 섹션, 행과 다이얼로그

#### 결과

기능별 파일의 책임이 명확해졌고, 이후 변경에서 필요한 파일만 좁게 확인할 수 있게
됐다. AI에게 작업을 맡길 때도 핵심 규칙과 UI 수정 범위를 분리해 지시하기 쉬워졌다.

### 5.3 휴대폰과 iPad 반응형 UI 개선

#### 문제

휴대폰에서 잘 보이는 화면이 iPad에서는 지나치게 비어 보이거나, 반대로 작은 화면에서
점수 카드와 랭킹, 플레이 버튼이 한 화면 안에 안정적으로 들어오지 않는 문제가 있었다.

#### AI 활용

AI에게 다음 우선순위를 주고 레이아웃을 분석하도록 했다.

4. 태블릿에서는 콘텐츠가 과도하게 늘어나지 않도록 최대 너비를 제한해야 한다.
5. 화면 크기에 따라 여백, 글자 크기와 목록 개수를 조정해야 한다.

AI가 제안한 제약 조건과 비율을 코드에 반영하고, 개발자가 실제 화면의 인상과 조작

## 6. 대표 지시문

아래 문장은 작업 의도를 보존한 요약이다. 실제 작업에서는 관련 소스 코드, 오류 로그와
실행 결과를 함께 제공했다. 과거 대화 원문이 남아 있다면 최종 문서에는 원문 캡처나

> 일일 도전이 제출 중 실패하더라도 사용자의 시도와 점수가 사라지지 않게 상태를
> 분석하고, 이미 제출된 점수는 중복 처리되지 않도록 복구 흐름과 테스트를 만들어줘.

### 반응형 화면

> 휴대폰과 iPad에서 핵심 정보와 플레이 버튼이 잘 보이도록 현재 화면 구조를 확인하고,
> 기기 크기에 맞는 최대 너비와 간격을 적용해줘.

### 공모전 준비

> NAN 2026 공식 제출 요건을 확인하고 현재 게임과 저장소를 비교해 누락된 자료를
> 정리해줘. 핵심 게임 규칙은 승인 없이 변경하지 말고 문서와 검증부터 준비해줘.

## 9. AI 활용의 한계와 안전장치

### 9.1 런타임 AI 의존성이 없다

게임 플레이에 외부 생성형 AI API를 필수로 사용하지 않는다. 따라서 API 장애,
응답 지연, 비용 또는 모델 변경이 게임 한 판의 결과를 바꾸지 않는다.

### 9.2 핵심 규칙 변경을 제한한다

`docs/game_rules.md`에는 게임 규칙을 수정할 때 사용자 승인을 받아야 한다고 명시되어
있다. AI가 리팩터링이나 버그 수정을 수행할 때도 점수, 시간과 매칭 규칙을 임의로
바꾸지 않도록 경계를 둔다.

### 9.3 코드를 실행해 검증한다

AI가 작성한 코드가 문법적으로 자연스러워 보여도 실제로 동작한다는 보장은 없다.
정적 분석, 테스트와 빌드를 실행하고, UI와 조작감은 실제 기기에서 사람이 확인한다.

### 9.4 비밀 정보는 저장소에서 제외한다

Supabase URL과 익명 키는 `.env` 또는 빌드 환경 값을 사용한다. `.env`는 `.gitignore`에
포함되어 공개 Git 기록에 올라가지 않도록 관리한다.

### 9.5 AI 결과를 그대로 저작물로 제출하지 않는다

AI가 만든 설명과 코드는 사람이 현재 게임과 일치하는지 확인하고 수정한다. 외부
에셋은 생성 도구의 이용 약관과 상업 이용 권한을 별도로 확인한다.

실제 기록과 다른 예시 문장을 그대로 사용하면 안 된다.

## 12. AI 활용 기술 문서에 추가하면 좋은 시각 자료

- 사람과 AI의 역할 분담 다이어그램
- 문제 제시 → 저장소 분석 → 코드 수정 → 테스트 → 사람 검토 흐름도
- `codex/daily-challenge-retry` 브랜치와 관련 커밋 캡처
- 일일 도전 복구 전후 상태 흐름도
- 리팩터링 전후 파일 구조 비교
- 자동 테스트 통과 화면
- Flutter analyze 결과 화면
- GitHub 커밋 이력 화면
- 실제 AI 대화 또는 프롬프트 캡처

터미널 캡처는 개인 키, 이메일, 환경 값과 로컬 경로가 노출되지 않도록 확인한다.

## 13. AI 활용에 대한 최종 요약 문장

Bee House에서 AI는 게임의 규칙을 대신 결정하거나 플레이 결과를 임의로 생성하지
않았다. 개발자가 핵심 경험과 품질 기준을 정의하고, OpenAI Codex를 이용해 저장소
분석, 구현, 리팩터링, 실패 복구, 테스트와 문서화를 반복했다. 그 결과는 공개된 전체
소스, 커밋 기록, 테스트와 데이터베이스 migration으로 확인할 수 있다. AI를 단순한
아이디어 생성 도구가 아니라, 사람이 지휘하고 실행 결과로 검증하는 제작 파트너로
활용한 것이 이 프로젝트의 AI 활용 방식이다.
